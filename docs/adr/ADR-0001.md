# ADR-0001: フロントエンドに Inertia.js + Vue 3 を採用する

## ステータス
承認済み

## 背景
Laravel をバックエンドとして採用した上で（ADR-0002）、フロントエンドの実装方法を選定する必要があった。
要件として「詳細なグラフ・ダッシュボード」「マルチカラムレイアウト」「フォーム多用」「PC ブラウザ中心」が挙げられており、
Laravel との統合コストが低く、型安全で長期保守しやすい選択が求められた。

## 決定
**Inertia.js v2 + Vue 3 + TypeScript + Vite** を採用する。

## 代替案

| 案 | 概要 | 却下理由 |
|---|---|---|
| Blade + Alpine.js | Laravel 標準のテンプレートエンジン + 軽量 JS | グラフ（Chart.js）・タグ多選択など複雑なインタラクションには Alpine.js では非力。コンポーネント化・型安全性も劣る |
| Next.js（SPA 分離） | React + API サーバーとして Laravel を使う | フロント・バックエンドのリポジトリ分離コスト大。認証フロー（セッション or JWT）の追加実装が必要 |
| Nuxt.js（SSR） | Vue + Nuxt で SSR | API サーバーとのフルスタック分離が必要で、Laravel との統合にオーバーヘッドが発生する |
| Livewire | PHP 中心のリアクティブ UI | 複雑なフォームロジックやグラフ表示を PHP 側で制御するのは不自然。TypeScript による型安全性が得られない |

## 結果
- **ポジティブ**: Laravel の `inertia-laravel` パッケージで Server-side routing（`Route::get`）をそのまま利用できる。SPA として動作しつつ、認証・セッション・CSRF をすべて Laravel に任せられる。Vue 3 の Composition API + TypeScript で型安全なフロントエンドが実現できる。
- **ネガティブ**: Inertia を知らない開発者には学習コストがかかる。SSR（SEO）は Inertia SSR オプションで対応可能だが、本プロジェクトではスコープ外。
- **制約**: 本 ADR を覆す場合、全ページコンポーネントと Controller の `Inertia::render` 呼び出しを全面再実装する必要がある。
