# ADR-0002: バックエンドフレームワークに Laravel 11 を採用する

## ステータス
承認済み

## 背景
学習記録管理アプリのバックエンドとして、フレームワーク選定を行う必要があった。
要件として「マルチユーザー対応の認証」「学習記録の永続化」「集計クエリ（ダッシュボード用）」「将来的な自己ホスト（Raspberry Pi 等）への対応」が挙げられた。
長期的に保守しやすく、エコシステムが豊富な選択が求められた。

## 決定
**Laravel 11（PHP 8.3）** を採用する。

## 代替案

| 案 | 概要 | 却下理由 |
|---|---|---|
| Supabase（BaaS）+ Next.js | バックエンドなし構成 | ベンダーロックイン。自己ホスト時に Supabase 自体のコンテナも管理が必要で複雑 |
| Express.js + TypeScript | Node.js サーバー | PHP/Laravel より本格的な認証・ORM・バリデーション機能を自前実装するコストが大きい |
| Django（Python） | Python フレームワーク | PHP の方が共有サーバー等の自己ホスト環境での互換性が高い。Laravel の ORM・マイグレーション体験が優れている |
| Ruby on Rails | Ruby フレームワーク | メモリ消費量が多く Raspberry Pi への自己ホストには不向き |

## 結果
- **ポジティブ**: Eloquent ORM・マイグレーション・認証（Breeze）・バリデーション・ファイルストレージが標準搭載されており、追加インフラなしで全機能を実装できる。PHP-FPM による低メモリ消費で Raspberry Pi 等の小型サーバーにも適する。
- **ネガティブ**: PHP への学習コストがある場合、最初の立ち上げに時間がかかる。
- **制約**: PHP 8.3 以上を要求する。本 ADR を覆す場合、全 Controller・Model・マイグレーションの再実装が必要。
