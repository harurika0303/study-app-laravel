# ADR-0009: テスト戦略として PHPUnit（Feature + Unit）+ Vitest を採用する

## ステータス
承認済み

## 背景
個人プロジェクトかつ AI 実装フェーズであるが、学習記録という個人データを扱う性質上、
コアロジックの破壊的変更を検知する仕組みが必要。
また、認証・コンテンツ追加・セッション記録といった主要ユーザーフローの回帰テストも求められる。

## 決定
**PHPUnit（Feature テスト + Unit テスト）+ Vitest（フロントエンド Composable テスト）** を採用する。
E2E テスト（Playwright 等）は現フェーズではスコープ外とする。

### 優先テスト対象

**Feature テスト（PHPUnit）**
- 認証フロー（ログイン・ログアウト・パスワードリセット）
- コンテンツ CRUD（作成・更新・削除・認可チェック）
- セッション記録（作成・削除・ステータス自動遷移）
- タグ CRUD

**Unit テスト（PHPUnit）**
- `DashboardService` の集計ロジック
- `ContentQueryService` のフィルタ・ソートロジック

**Vitest**
- `useSessionTimer.ts`（localStorage 操作ロジック）
- ユーティリティ関数

## 結果
- **ポジティブ**: `RefreshDatabase` を使うことで毎回クリーンな DB でテストできる。Factory でテストデータを素早く生成できる。
- **ネガティブ**: E2E テストがないため、フロントエンドの描画バグは手動確認が必要。
